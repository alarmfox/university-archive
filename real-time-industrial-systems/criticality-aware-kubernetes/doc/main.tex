\section*{Introduzione}
In un cluster Kubernetes, la Kubelet si occupa di gestire lo stato di un nodo worker
scambiando informazioni con la parte di controllo.
Lo scopo del progetto è quello di analizzare il funzionamento della Kubelet
per verificare se questa può essere estesa per schedulare
applicazioni \textit{soft real-time}. In particolare, l'analisi mira a ricavare i fattori principali che influenzano la latenza per lo startup dei pod in Linux.

\section{Kubenernetes}\label{section:kubernetes}
Kubernetes\cite{KubernetesDocs} è un orchestratore di applicazioni containerizzate che automatizza le attività di \textit{deploy} e configurazione in ambito cloud. Un cluster Kubernetes è formato da:
\begin{itemize}
    \item \textbf{Data plane}: contiene i \textbf{worker node} che eseguono le applicazioni in \textit{pod}. Un pod è la più piccola unità installabile in Kubernetes. Il componente che si
    occupa di gestire lo stato del nodo e le sue risorse è detto Kubelet che 
    comunica con il \textit{control plane} per ricevere istruzioni e fornire informazioni sullo stato dei pod;
    \item \textbf{Control plane}: è un gruppo di nodi che fornisce delle API all'utente e coordina l'intero cluster. Le richieste ricevute all'utente
    vengono trasmesse ai nodi su cui sarà eseguita l'applicazione;
\end{itemize}

L'architettura di un cluster generico è mostrata in \cref{fig:kube-arch}.
\begin{figure}[h]
    \centering
    \includegraphics[width=.9\textwidth]{figures/ch1/kubernetes_arch.png}
    \caption{Architettura di un cluster Kubernetes}
    \label{fig:kube-arch}
\end{figure}

\subsection{Control plane}
Il control plane si occupa di gestire il dataplane attraverso l'\textbf{API server} che si collega 
con i worker nodes attraverso \textbf{gRPC}. Al suo interno, lo stato dei nodi viene aggiornato 
con i messaggi comunicati dalla kubelet e memorizzati in \textbf{etcd}: un database chiave-valore distribuito. 
Il control plane usa il \textbf{kube-scheduler} per selezionare il nodo più adatto al workload da installare. Tra i fattori tenuti in considerazione ci sono:
\begin{itemize}
    \item Vincoli imposti dal workload: un'applicazione può decidere influire sulla scelta del worker node specificando sia vincoli sull'host che sulle risorse richieste;
    \item Stato del cluster: lo scheduler cerca di bilanciare le risorse utilizzate dai nodi worker e non 
    assegna pod ad un nodo non \textit{healthy};
\end{itemize}

\subsection{Data plane}
Il data plane è formato da nodi worker che si occupano di eseguire i workload commissionati dal master sottoforma di 
pod. Un pod è un gruppo di container che viene gestito in maniera atomica dai componenti del cluster. Ogni worker node ha due componenti principali:
\begin{itemize}
    \item \textbf{Kubelet}: agente che si interfaccia con il control plane, si occupa di gestire fisicamente le risorse dell'host e di interagire con il \textbf{CRI (Container Runtime Interface)} che astrae la tecnologia di creazione del container. Nel progetto tutti gli esperimenti usano container eseguiti attraverso \textbf{containerd};
    \item \textbf{Kube-proxy}: componente che accetta il traffico di rete degli utenti reali;
\end{itemize}

\subsubsection{Kubelet: ciclo di controllo}
Il control plane di kubenetes si occupa di verificare continuamente lo stato del cluster affinchè sia
quello desiderato dall'utente. Quando viene effettuata una modifica dall'esterno o si verifica un evento nel worker node, il control plane calcola le \textbf{mutazioni} da applicare allo stato corrente per raggiungere quello desiderato. Le mutazioni sono messaggi diretti alla kubelet che le traduce in 
azioni sui pod.

 La Kubelet funziona eseguendo un ciclo di sincronizzazione applicando aggiornamenti che arrivano da 3 canali (\cref{fig:worker-node}):
\begin{itemize}
    \item File di configurazione;
    \item Richieste http;
    \item API server (richieste dal control plane);
\end{itemize}

\begin{figure}[H]
    \centering
    \includegraphics[width=.95\textwidth]{figures/ch1/kubelet_syncloop.png}
    \caption{Struttura di un worker node in kubernetes}
    \label{fig:worker-node}
\end{figure}

Inoltre, periodicamente ogni secondo controlla che tutti i pod per assicurarsi che lo stato corrente corrisponda con quello desiderato (\cref{fig:syncloop}).
Viene qui riportata la \textit{signature} del metodo che gestisce il ciclo dapprima descritto.
\begin{verbatim}
    func (kl *Kubelet) syncLoop(ctx context.Context, 
            updates <-chan kubetypes.PodUpdate, 
            handler SyncHandler) 
\end{verbatim}
Nel dettaglio i tre parametri d'ingresso sono:
\begin{itemize}
    \item \begin{verbatim} type Context interface\end{verbatim} L'interface contiene una Deadline, un channel per la cancellazione, errori ed evenutali valori di gestione utente.
    \item \begin{verbatim} type PodUpdate struct \end{verbatim} La struct contiene un vettore di pod da aggiornare, il tipo di operazione da effettuare ed il nome di tale update.
    \item \begin{verbatim} type SyncHandler interface\end{verbatim} L'interface contiene dei metodi per gestire; l'eliminazione, la riconfigurazione, la podAddition ecc, ecc.
\end{itemize}

\subsubsection{Kubelet in Linux}
La kubelet controlla le risorse hardware attraverso i \textbf{cgroups}. Un cgroup è un meccanismo di isolamento che permette di limitare e controllare le risorse utilizzate da un gruppo di processi. La kubelet partiziona le risorse dell'host in cui si trova in $3$ gruppi:
\begin{itemize}
    \item \textbf{kubepods}: risorse dedicate ai pod. Un pod è l'unità minima schedulabile da kubernetes composta da un gruppo di container che condividono le stesse risorse;
    \item \textbf{system-resources}: utilizzato per i processi che non fanno parte di kubernetes;
    \item \textbf{kube-resources}: risorse allocate per tutti i processi collegati a kubernetes sull'host quali container runtime, la stessa kubelet ecc;
\end{itemize}
A sua volta le risorse dedicate ai pod vengono raggruppate in $3$ classi di Quality of Service:
\begin{itemize}
    \item \textbf{Guaranteed}: un pod viene considerato \textit{guaranteed} se tutti i suoi container hanno richieste/limiti specificati sia per CPU che per memoria e sono uguali per ogni container;
    \item \textbf{Burstable}: un pod viene considerato \textit{burstable} se almeno un container del pod ha una richiesta per una risorsa tra CPU e memoria;
    \item \textbf{Best effort}: un pod viene considerato \textit{best effort} se nessun container presenta una richiesta per CPU e memoria;
\end{itemize}

\subsection{Container in Linux} \label{section:linux_container}
Un container è un'applicazione isolata che si basa sulla condivisione del kernel e per questo motivo è molto facile da gestire. Per garantire disaccoppiamento, la Kubelet non crea container interagendo direttamente con il Kernel, ma, come mostrato in \cref{fig:kubelet_2_containerd}, utilizza un'interfaccia chiamata \textbf{Container Runtime Interface (CRI)} implementata da uno specifico componente: \textbf{Container Runtime}.

\begin{figure}[h]
    \centering
    \includegraphics[width=.8\textwidth]{figures/ch1/containerdimg.png}
    \caption{Architettura completa dell'interazione tra containerd e kubelet}
    \label{fig:kubelet_2_containerd}
\end{figure}

In questo progetto, il container runtime utilizzato è \textbf{containerd}. In particolare, la kubelet utilizza un componente detto \textbf{cri-containerd} che si occupa di usare le API \textbf{gRPC} messe a disposizione da containerd. Si noti anche che, dalla \cref{fig:kubelet_2_containerd}, la gestione di un pod con $n$ container comporta effettivamente la creazione di $n+1$ container in quanto la kubelet necessita di un un container speciale detto \textbf{Sandbox} che utilizza l'immagine \textbf{pause}. Questa speciale applicazione è composta da codice \textit{assembly} e ha il compito di tenere attivo il \textbf{network namespace} del pod\cite{WhatIsPauseContainer}. I namespace in Linux vengono cancellati quando nessun processo ne fa parte e, pertanto, per evitare downtime quando un container viene eliminato è necessario avere quest'applicazione\cite{WhyPauseContainer}\cite{ManUnshare}.

\paragraph{Contaienrd} Containerd\cite{Containerd} è un \textbf{high-level} container runtime scritto in Go che crea l'astrazione di container. Il suo compito principale è quello di gestire il ciclo di vita di un container comunicando con un \textbf{low-level} container runtime. Quest'ulteriore astrazione è stata realizzata per tenere separata la particolare implementazione del container dal suo utilizzo ed è possibile grazie allo standard \textbf{OCI}\cite{OpenContainerInitiative} che specifica il formato di un immagine. Containerd utilizza come default low-level runtime \textbf{runc} che si occupa di creare container in Linux.

\paragraph{Runc} Runc si occupa di creare un container attraverso le system-call di Linux. In particolare, parte dalla creazione di un processo e lo configura in base al tipo di richiesta ricevuta attraverso i namespace con delle operazione di I/O (leggendo e scrivendo in dei file virtuali). Il flusso di creazione di un container prevede l'utilizzo di una doppia \textit{fork} (è necessario eseguire le system-call \textbf{clone} e \textbf{unshare}): la prima viene utilizzata per creare il processo container la seconda per assicurarsi che il container non venga eliminato se runc termina e viceversa (il container non deve diventare uno \textit{zombie} nel caso in cui runc termini).

Di seguito, è riportato un esempio\cite{RuncContainerConfig} di configurazione in \textbf{Go} per la creazione di un container che utilizza le API di basso livello in \textbf{C}.

Di particolare interesse è la funzione \textbf{nsexec} che viene utilizzata proprio per creare un container. Dall'analisi del codice, emerge che le systemcall utilizzate per creare un container sono:
\begin{itemize}
    \item \textbf{clone()}: crea un nuovo processo. Si tratta di una versione a controllo più della \textbf{fork} in quanto l'utente può controllare i namespace nei quali viene messo il nuovo processo e il suo spazio di indirizzamento virtuale \cite{ManClone};
    \item \textbf{unshare()}: consente al processo corrente di separare parte del suo contesto esecutivo\cite{ManUnshare};
    \item \textbf{pivot\_root()}: cambia la cartella root per il processo chiamante;
    \item \textbf{chdir()}: cambia la \textit{current working directory} del processo chiamante;
    \item \textbf{close()}: chiude il \textit{file descriptor} passato in ingresso;
    \item \textbf{setns()}: prende in ingresso un \textit{file descriptor} che punta ad un namespace. Associa il processo chiamante al namespace indicato\cite{ManSetns};
    \item \textbf{mount()}: \textit{monta} il filesystem in ingresso sulla direzione specificata dal processo chiamante;
    \item \textbf{chmod(), chown()}: configurano i permessi sui filesystem \textit{montati}.
    \item \textbf{sethostname(), setdomainname()}: configura il nome host e il dominio per identificare il container sulla rete;
    \item \textbf{setrlimit()}:  configura i limiti applicando la configurazione della richiesta, come mostrato nell'esempio \ref{lst:runc_config};
    \item \textbf{execve()}: esegue il comando specificato dal processo;
\end{itemize}

\section{Analisi latenza per la creazione di un pod}

\subsection{Problema} Il problema che si vuole affrontare nel progetto è quello di proporre un limite superiore (anche teorico) al tempo di creazione e startup di un container in Kubernetes. In particolare, nel progetto ci si è focalizzati sulla latenza generata dalla kubelet (\cref{fig:pod_latency} sezione in verde) che dipende interamente dal nodo worker e dalla tecnologia con cui sono implementati i container.

\paragraph{Perchè Kubernetes non è adatto a schedulare task hard real-time?} Kubernetes e containerd sono due applicazioni scritte Go\cite{Golang}. Go è un linguaggio di alto livello dotato di un \textbf{runtime} e \textbf{garbage collected}. Progettando accuratamente le risorse dell'applicazione è possibile disabilitare il garbage collector\cite{GoGarbageCollector} e procedere con un'allocazione manuale attraverso il \textbf{C-binding} messo a disposizione con \textbf{CGO}. Il runtime è dotato di uno scheduler \textbf{M:N} fair\cite{GolangScheduler} che si occupa di gestire $m$ go-routines su $n$ processi creati in base al numero di core. Per bilanciare il carico sui processori, viene utilizzata la tecnica del \textbf{work stealing}\cite{GolangSchedulerDeepDive} con un time slice di $10ms$\cite{GolangScheduler}. Questa implementazione rende Go non adatto per applicazione \textbf{hard real-time}.

\textbf{Kubectl} è un programma CLI che consente di interagire con un cluster kubernetes attraverso un file di configurazione. Quando si crea un deployment, viene contattato il \textbf{control plane} che si preoccupa di eseguire la richiesta interagendo con le kubelet dei \textbf{worker node}.

\subsubsection{Extreme value analysis su tempo di creazione di un pod end-to-end}
La Extreme Value Theory\cite{Evt} si occupa di stimare un WCET probabilistico analizzando le code delle distribuzioni ed è stata utilizzata perché:
\begin{itemize}
    \item Non assume che le distribuzioni siano normali;
    \item Si concentra su eventi critici;
\end{itemize}
Per essere applicata, questa teoria ha bisogno di $3$ ulteriori ipotesi sulle distribuzioni:
\begin{itemize}
    \item \textbf{Stazionarietà}: Un esecuzione piatta dei task con varianza costante, non ci sono task che cambiano drasticamente i loro tempi di esecuzione;
    \item \textbf{Short-Range independence}: Supponiamo di non avere tempi di accesso dipendenti dalla cache;
    \item \textbf{Long-Range independence}: L'esecuzione dei task non dipendono dal test specifico;
\end{itemize}

Sulle ipotesi, la EVT afferma che la distribuzione creata a partire dal valore massimo converge alla \textbf{Generilized Extreme Distribution (GEV)} \cref{eq:gev} che può avere $3$ forme in base al parametro $\xi$:
\begin{itemize}
    \item ($\xi = 0$): distribuzione di Gumbel;
    \item ($\xi > 0$): distribuzione di Frechet;
    \item ($\xi < 0$): distribuzione di Weibull;
\end{itemize}

\begin{equation}
    F(x) = \exp{\big(-\big(1 + \xi \frac{x-\mu}{\sigma}\big)^{-\frac{1}{\xi}}}\big)
    \label{eq:gev}
\end{equation}

In accordo con quanto detto in \cref{section:results}, i fit più interessanti (riportati in \cref{tab:fit_data}, grafici $d, e, g$ della figura \cref{fig:qqplot-gev}) risultano essere quelli in cui viene utilizzato lo scheduler FIFO. Nei casi con fit migliore $\xi < 0$, quindi ci si trova in presenza di una distribuzione di \textbf{Weibull} già utilizzata per gli studi di \textit{reliability} e per modellare il \textbf{Mean Time to Failure (MTTF)} in sistemi critici\cite{WeibullUses}.

\section{Limite superiore al tempo di startup di un pod}
\subsection{Modello teorico per il tempo di startup di un pod}
In base alla descrizione fornita in \cref{section:kubernetes}, Kubernetes può essere modellato come un sistema mixed criticality in cui ci sono applicazioni critiche (indicate come pod) con vincoli temporali sulla latenza. In questo caso, come mostrato in \cref{fig:pod_latency}, la latenza totale per la creazione di un pod $L_{pod}$ è composta da due parti:
\begin{equation*}
    L_{pod}(S) = L_{cp}(S_{cp}) + L_{wn}(S_{wn}) + \delta
\end{equation*}
in cui con $L_{cp}(S)$ si indica la latenza generata dal \textbf{control plane} del cluster e con $L_{wn}(S)$ la latenza della worker node. Con $S_{wn}$, si indica lo stato del nodo worker su cui sarà avviato ed installato il pod, con $S_{cp}$ lo stato del control plane. Come anticipato, il progetto verte sull'analisi della kubelet e pertanto sarà trascurato $L_{cp}(S_{cp})$ assumendolo già limitato superiormente. Infine, con $\delta$ si indica la latenza introdotta dalla comunicazione di rete per l'interazione tra kubelet e master.

Analizzando il codice sorgente di kubelet, runc e containerd gli eventi che generano latenza, evidenziati in \cref{fig:pod_latency}, sono:
\begin{enumerate}
    \item $t_w$: tempo di attesa della richiesta in coda;
    \item $t_i$: preparazione dell'immagine del container da utilizzare. Questo tempo include anche il tempo di \textit{download} dell'immagine;
    \item $t_{runc}$: esecuzione delle system-call sul sistema operativo host da runc per l'effettiva creazione del pod\footnote{Come specificato in \cref{section:linux_container}, la creazione di un pod può includere più container};
\end{enumerate}

\begin{figure}[h]
    \centering
    \includegraphics[width=.8\textwidth]{figures/ch3/latency.png}
    \caption{Analisi latenza di un pod in kubernetes con focus sul worker node}
    \label{fig:latency-detail}
\end{figure}

In \cref{eq:pod_latency}, è riportata l'espressione per stimare la latenza per la creazione di un pod. Tutti i membri della relazione dipendono dallo stato del nodo.
\begin{equation}
    L_{wn}(S_{wn}) =  t_w(S_{wn}) +  t_i(S_{wn}) + t_{runc}(S_{wn})
    \label{eq:pod_latency}
\end{equation}

\paragraph{Assunzioni}
Per le applicazioni real-time, si ipotizza che un pod possa contenere solo un container. Questo è dovuto alle modalità con cui la kubelet si interfaccia con il master: un pod con più applicazioni potrebbe essere ricevuta con più richieste separate oppure una sola delle applicazioni potrebbe bloccarsi richiedendo la ricreazione di un solo container.

\subsubsection{Stato e limiti del modello} I tempi indicati in \cref{eq:pod_latency} dipendono fortemente dallo stato del sistema. In ipotesi semplificata, lo stato del sistema può essere modellato con $2$ variabili:
\begin{itemize}
    \item $n$: numero di processi in esecuzione nel sistema;
    \item $n_q$: numero di richieste in coda alla Kubelet;
\end{itemize}

Il limite principale di questa formulazione teorica è la codifica dello stato. Infatti, il kernel Linux è molto complicato e non è possibile teorizzare in maniera esaustiva tutto il suo stato.

\subsubsection{Analisi $t_i(S_{wn})$}
Questo tempo coinvolge tutte le attività che servono a preparare il container e coinvolge sia operazioni su fileystem (copia dell'immagine eseguibile in memoria, modellato con una costante $c$) sia di rete (eventuale download dell'immagine, modellato con la variabile $t_d$). In entrambi i casi, $t_i$ si può modellare con:
\begin{equation*}
    t_i = \begin{cases}
    c & \text{se l'immagine è presente già nel registry locale} \\
    t_d + c & \text{altrimenti}
    \end{cases}
\end{equation*}

Nel caso di containerd, questo tempo può essere $c$ se si assicura che l'immagine del container sia sempre presente sul nodo utilizzando il meccanismo dei tag. Infatti, se si richiede la creazione di un container con tag specifici, containerd non effettuerà alcuna richiesta in rete se questa è già presente in locale.

\subsubsection{Analisi $t_{runc}(S_{wn})$} Dall'analisi del file \textit{nsexec.c} in \cite{Runc}, risulta che per creare un container servono $11$ system-call (elencate in \cref{section:linux_container}). Con l'ipotesi del \textit{single container pod}, si può stimare la latenza generata da \textit{runc} come:
\begin{equation*}
    t_{runc}(S_{wn}) = 2 \cdot m \cdot \max (T_{sys})
\end{equation*}
dove con $m$ si indica il numero di system-call effettuate da runc (in questo caso $m=11$) e con $T_{sys}$ il tempo per effettuare una system-call. Inoltre, nell'ipotesi di WCET, si moltiplica per $2$ perchè bisogna tenere conto dell'eventuale creazione del container \textit{pause} descritto in \cref{section:linux_container}.

\paragraph{Perchè non si usa una soluzione real-time come Xenomai?} Xenomai è una soluzione dual kernel in cui si trattano in maniera esplicitamente diversa i task real-time attraverso EVL core. Questa soluzione al momento non supporta i namespace e quindi non può essere usata per creare container.

\subsubsection{Analisi $t_w(S_{wn})$}
$t_w(S_{wn})$ si indica l'intervallo di tempo che una richiesta gRPC in arrivo dal control plane spende in coda al worker node prima di essere servita. Mettere un limite superiore al tempo di attesa in una coda richiede:
\begin{itemize}
    \item applicazione di politiche di rate limiting;
    \item modifica delle politiche di scheduling;
\end{itemize}
La teoria delle code utilizza i valori medi per stimare i tempi medi di attesa delle richieste, ma per avere un WCET bisogna utilizzare la Extreme Value Theory. Inoltre, non sono verificate le ipotesi di teorema di limite centrale.

\paragraph{Rate limiting: algoritmo fixed window}
In questo approccio, viene definita una finestra di tempo di durata predefinita in cui è possibile server un numero limitato di richieste in un certo intervallo temporale. Se il numero di richieste supera una soglia predefinita (limite di rate), le richieste eccedenti possono essere respinte, ritardate o gestite in qualche modo, come mostrato in \cref{fig:fixed_window}\cite{FixedWindowAlgorithm}. I parametri principali dell'algoritmo sono:
\begin{itemize}
    \item \textbf{maxCount}: numero massimo di richieste nella coda;
    \item \textbf{windowLength}: durata dello slot temporale. Quando termina uno slot, viene effettuato un reset del numero di richieste presenti nel sistema;
\end{itemize}

\paragraph{Scheduling: deferrable server}
Il deferrable server fornisce un test di schedulabilità teorico in base ai suoi parametri e il numero di task nel sistema. Questa soluzione potrebbe essere introdotta per garantire ad ogni richiesta di pod real-time una deadline.

\begin{equation*}
    U_p \leq n \big[\big(\frac{U_s + 2}{2U_s + 1}\big)^{1/n} - 1\big]
\end{equation*}

Questa soluzione è collaudata con lo scheduler rate-monotonic già implementato nel Kernel Linux attraverso Sched-FIFO utilizzato per gli esperimenti.
